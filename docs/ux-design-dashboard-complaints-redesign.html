Design Notes

Building on Requirements

This design implements all 11 acceptance criteria from the Requirements step. Key design decisions:

Dashboard (/dashboard)

  • Summary cards kept as-is — they provide quick context
  • Complaint list simplified: 5 rows, no filters, no sorting controls
  • "View all complaints →" link in both the section header and a footer bar for discoverability
  • Quick action buttons (copy/email) removed from dashboard — they belong on the full Complaints page

Complaints Page (/dashboard/complaints)

  • Search bar is prominent at the top — most common action for finding a specific complaint
  • Filters below search in a single row: status, category, recipient type, date range, sort
  • Table layout with columns: Recipient, Category, Date, Status (dropdown), Response (dropdown), View link
  • Status and Response are inline <select> dropdowns — change triggers immediate API update (no save button needed)
  • Response column shows "—" for complaints that haven't been sent yet (no dropdown)
  • Overdue badge shown next to response status when follow-up date has passed
  • Pagination at bottom: Previous/Next + "Page X of Y" + "Showing X-Y of Z"

Nav Bar

  • "Complaints" tab added between Dashboard and Recipients
  • Active state uses the same bg-navy-800/5 pill style as existing tabs
  • "New Complaint" text link removed (per earlier bug fix) — CTA button remains

States Not Shown (but required)

  • Empty state: "No complaints yet" with CTA to create first complaint (reuse existing EmptyState component)
  • No results: "No complaints match your search/filters" with clear filters button
  • Loading: Skeleton rows matching the table structure
  • Error: Same error card pattern as dashboard, with retry button
  • Mobile: Stack into card layout (same pattern as existing complaint-list.tsx mobile view), search/filters collapse behind toggle
