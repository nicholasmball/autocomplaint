Extend Scraper to Councils & Regulators

API-only change — no UI modifications required

API Contract Change

Building on the Requirements step. The only user-facing change is the API request body.

Before

POST /api/admin/scraper
{
  "companyId": "uuid",
  "dryRun": true,
  "useJsRendering": true
}

After

POST /api/admin/scraper
{
  "table": "councils",
  "recipientId": "uuid",
  "dryRun": true,
  "useJsRendering": true
}

table — 'companies' | 'councils' | 'regulators' (default: 'companies')

companyId renamed to recipientId for clarity (companyId still accepted for backward compat)

Domain Resolution Logic

1

Check website field

Councils and regulators have a website column. If populated, use it directly as the base domain.

2

Fallback: guess domain from name

If no website, generate slug from name and try domain patterns based on table type:

Companies

{slug}.co.uk
{slug}.com

Councils

{slug}.gov.uk
{slug}.co.uk

Regulators

{slug}.org.uk
{slug}.gov.uk
{slug}.co.uk

3

Try contact page paths

Same as current: /complaints, /contact, /contact-us, /help/complaints, /help/contact

4

Extract email with Claude

Reuse existing extractEmail() — no changes needed.
